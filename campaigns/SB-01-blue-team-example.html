SB-01 Blue Team Detection Mapping Example

이 페이지는 팀원들이 2페이지를 작성할 때 참고할 수 있는 예시다. 핵심은 Techniques Used를 단순 설명으로 끝내지 않고, 각 행위가 실제 기업형 환경의 어느 로그와 아티팩트에 남는지까지 연결하는 것이다.

Example Scope

Technique 왜 예시로 골랐는가 핵심 로그 소스 방어자 관점 질문
T1190 Jenkins CLI 취약점 악용 단계로, 공격 시작점과 취약 서비스 악용 흔적을 설명하기 좋다. Jenkins access/system/container log, CLI request evidence 어떤 계정/토큰이 Jenkins CLI를 호출했고, 어떤 파일 접근 시도가 있었는가?
T1021.004 Jenkins에서 App 서버로 이동하는 측면이동 단계라서 auth.log와 네트워크 흐름이 명확하다. App Server /var/log/auth.log, Jenkins build log, VPC Flow Log deploy 계정 SSH 접속이 정상 배포인지, 비정상 측면이동인지 어떻게 구분할 것인가?
T1213.006 최종 피해와 연결되는 DB 데이터 수집 단계라서 PostgreSQL 로그와 업무 데이터 영향도를 설명하기 좋다. PostgreSQL connection/query log, App API log, DB host auth log 예약/고객 데이터 조회가 정상 API 호출인지, 비정상 대량 조회인지 어떻게 판단할 것인가?

T1190 - Exploit Public-Facing Application

2024 PipelineKey에서 BreadKimchi는 탈취한 Jenkins API Token으로 Jenkins CLI에 인증한 뒤, CVE-2024-23897 파일 읽기 취약점을 이용해 Jenkins 내부 파일 접근 가능성을 확인했다.

구분 구체 위치 수집 방법 관찰할 필드/패턴 분석 의미
Jenkins HTTP/CLI 요청 Jenkins controller container log
docker logs sb-jenkins		 Docker log driver 또는 Filebeat container input CLI endpoint 접근, 비정상 command, 인증 사용자, source IP Jenkins CLI가 평소 배포 흐름이 아닌 조사/파일 읽기 목적으로 호출됐는지 확인한다.
Jenkins system log /var/jenkins_home/logs/
/var/jenkins_home/logs/tasks/		 Jenkins home volume을 Filebeat path input으로 수집 CLI command, user id, authentication event, warning/error API Token을 사용한 CLI 접근과 Jenkins 내부 처리 이벤트를 연결한다.
취약 파일 읽기 시도 Jenkins CLI command argument
예: @/var/jenkins_home/...		 Jenkins log + BAS 실행 로그 + auditd 보강 @/var/jenkins_home, credentials, .ssh, secrets 단순 로그인과 달리 Jenkins controller 내부 파일 접근을 시도한 정황으로 본다.
보강 아티팩트 Jenkins host auditd
/var/log/audit/audit.log		 auditd rule로 Jenkins home 주요 경로 read 이벤트 수집 파일 경로, uid/gid, process, syscall, timestamp Jenkins 자체 로그가 부족할 때 파일 접근 증적을 보완한다.

ELK Query Draft

event.dataset:("jenkins" or "docker.container")
and message:(*@/var/jenkins_home* or *jenkins-cli* or *connect-node* or *help*)

Investigation Pivot

  • 이 이벤트가 보이면 먼저 Jenkins API Token을 사용한 계정과 source IP를 확인한다.
  • 그 직전 개발자 단말/VPN 접속 흔적을 확인한다.
  • 그 직후 App 서버로 SSH 접속이 발생했는지 확인한다.

T1021.004 - Remote Services: SSH

BreadKimchi는 Jenkins 내부에서 확인한 배포 credential 및 관련 단서를 바탕으로 App 서버의 deploy 계정에 SSH로 접근했다. 이 단계는 Jenkins 침해가 App 서버 측면이동으로 이어졌는지 판단하는 핵심 지점이다.

구분 구체 위치 수집 방법 관찰할 필드/패턴 분석 의미
App 서버 SSH 인증 /var/log/auth.log Elastic Agent System integration 또는 Filebeat system module Accepted publickey for deploy, source IP, user, sshd pid deploy 계정으로 누가 App 서버에 접속했는지 확인한다.
Jenkins 배포 실행 로그 /var/jenkins_home/jobs/deploy-to-staging/builds/*/log Jenkins home build log path 수집 ssh deploy@..., build number, executor, stage name App 서버 SSH 접속이 정상 배포 Job에서 발생한 것인지 비교한다.
App 서버 네트워크 흐름 AWS VPC Flow Log
App ENI 기준		 VPC Flow Logs -> CloudWatch/S3 -> Logstash/Filebeat srcaddr=Jenkins private IP, dstport=22, action=ACCEPT Jenkins에서 App 서버로 SSH 연결이 실제 네트워크 레벨에서 있었는지 확인한다.
명령 실행 흔적 App 서버 auditd
/var/log/audit/audit.log		 auditd execve rule + Filebeat whoami, ls, cat, tar, scp, psql SSH 접속 후 단순 배포인지, 탐색/수집 행위로 이어졌는지 판단한다.

ELK Query Draft

host.name:"sb01-app"
and event.dataset:"system.auth"
and message:"Accepted publickey for deploy"

Investigation Pivot

  • 접속 source IP가 Jenkins private IP인지, 개발자 단말 또는 외부 IP인지 확인한다.
  • 같은 시간대 Jenkins build log에 정상 배포 Job이 있었는지 확인한다.
  • SSH 직후 .env, docker-compose.yml, DB 접속 행위가 있었는지 확인한다.

T1213.006 - Data from Information Repositories: Databases

BreadKimchi는 App 서버에서 확인한 PostgreSQL 접속 정보를 사용해 DB 서버의 고객 예약 데이터를 수집했다. 이 단계는 단순 시스템 침해가 실제 데이터 영향으로 전환되는 지점이다.

구분 구체 위치 수집 방법 관찰할 필드/패턴 분석 의미
PostgreSQL 접속 로그 DB 서버 PostgreSQL log
/var/log/postgresql/postgresql-16-main.log		 Filebeat custom log input 또는 PostgreSQL integration database, user, source IP, connection authorized/disconnection App 서버에서 DB 서버로 접속한 시점과 계정을 확인한다.
PostgreSQL query 로그 DB 서버 PostgreSQL statement log
/var/log/postgresql/postgresql-16-main.log		 log_statement, log_connections 활성화 후 수집 SELECT, reservations, customers, COPY, 대량 조회 예약/고객 데이터 조회가 발생했는지 확인한다.
App API 로그 App 서버
/var/log/spacebar-booking/api/app.log		 Filebeat path input, JSON parsing 가능 예약 생성/조회 endpoint, response status, client IP, request id 정상 API 조회인지, 서버 내부에서 직접 DB 접근한 것인지 구분하는 보조 근거다.
App 서버 명령 실행 App auditd 또는 shell/process log auditd execve rule + Filebeat psql, pg_dump, tar, gzip, openssl DB 데이터 수집과 압축/암호화 준비가 같은 세션에서 이어졌는지 확인한다.

ELK Query Draft

host.name:"sb01-db"
and event.dataset:"postgresql.log"
and message:(*reservations* or *customers* or *COPY* or *pg_dump*)

Investigation Pivot

  • DB 접속 source IP가 App 서버인지 확인한다.
  • 해당 DB 조회 직전에 App 서버에서 .env 또는 DB 설정 파일 접근이 있었는지 확인한다.
  • 조회 직후 App 서버에서 archive 생성 또는 외부 전송 이벤트가 있었는지 확인한다.
