SB-07 Detection Evidence

이 페이지는 OZZY PMS Chain의 Page 3 Detection Evidence이다. 목적은 공격을 차단했다는 주장이 아니라, 공격 이후 분석가가 ELK와 Windows 원본 아티팩트로 흐름을 다시 찾을 수 있음을 증명하는 것이다. 따라서 각 증거는 확인됨, 추론, 미확인을 분리해 기록한다.

Evidence Principle 이 페이지의 핵심은 "공격이 있었다"를 멋있게 말하는 것이 아니라, 어떤 KQL과 어떤 원본 아티팩트로 같은 결론에 다시 도달할 수 있는지를 남기는 것이다.

Evidence Summary

Evidence Status What It Proves Limit
HAD-005~HAD-009 loader artifact events 확인됨 DC에서 loader chain 관련 Windows artifact를 ELK에서 다시 조회할 수 있다. 이 자체가 manual mapping 내부 동작을 직접 기록한 것은 아니다.
HAD-009 USN Journal summary 확인됨 hgl_loader.exe, hgl_payload.enc, hgl_run.cmd 파일 lifecycle이 남았다. USN은 파일 변경 증거이며 process memory behavior 증거는 아니다.
HAD-005 Application Error / WER 확인됨 hgl_loader.exe가 실제 프로세스로 실행됐고 crash event가 남았다. crash만으로 LSASS access를 증명하지는 못한다.
HAC-001 correlation alerts 확인됨 단일 이벤트가 아니라 loader artifact와 Windows artifact를 시간창 내 incident 후보로 묶었다. alert는 원본 이벤트를 해석한 2차 산출물이다.
HAD-003 Sysmon ProcessAccess 미확인 설치 후 재현 시 hgl_loader.exe -> lsass.exe 직접 증거가 될 수 있다. 현재는 과거 실행 시점의 Event ID 10이 없어 직접 증명하지 않는다.

Evidence 2. Loader event.action pivot

event.action 기준으로 어떤 loader artifact가 들어왔는지 좁혀 보는 쿼리다.

campaign.id:"SB-07" and event.action:(
  "amcache_loader_artifact_found" or
  "shimcache_loader_artifact_found" or
  "application_loader_crash_observed" or
  "taskcache_loader_task_found" or
  "task_xml_loader_action_found" or
  "usn_loader_file_lifecycle_observed" or
  "loader_execution_log_found"
)

이 쿼리 결과는 DC에서 loader 실행 경로를 재구성하는 증거다. loader_execution_log_found는 custom execution artifact이고, application_loader_crash_observed, task_xml_loader_action_found, usn_loader_file_lifecycle_observed는 Windows 원본 로그/아티팩트 기반 요약 이벤트다.

Evidence 3. Correlation alert in Hanguel Alerts

단일 로그가 아니라 여러 증거를 시간창 안에서 묶어 incident 후보로 만든 결과를 확인한다.

Evidence 4. Direct Sysmon LSASS access check

이 쿼리는 아직 결과가 없어야 정상이다. 현재 문서에서는 hgl_loader.exe -> lsass.exe 직접 ProcessAccess를 확보했다고 주장하지 않는다. Sysmon 설치 이후 동일 계열 행위를 재현했을 때 직접 증거로 사용할 수 있다. 

event.action:"sysmon_lsass_process_access" and data.message_excerpt:*hgl_loader.exe*
event.code:10 and process.executable:*hgl_loader.exe and winlog.event_data.TargetImage:*lsass.exe
Evidence Boundary 이 항목이 없기 때문에 manual_mapping_inferred는 직접 증거가 아니라 추론으로 유지한다.

Analyst Reconstruction Chain

1. Patch ChainSearch HAW/HAP patch events
2. DC AccessPivot to WinRM/C$ events
3. Loader StagingFind task and file artifacts
4. Loader RunReview WER and run log
5. CorrelationConfirm HAC-001 alerts
6. BoundarySeparate direct evidence from inference

Capture Checklist

  1. Kibana Discover에서 HAD-005~HAD-009 조회 결과를 캡처한다.
  2. event.action별 loader artifact 결과를 캡처한다.
  3. Hanguel Alerts data view에서 sb07-loader-forensic-triad-012와 sb07-manual-mapping-inferred-014를 캡처한다.
  4. Windows Event Viewer에서 Application Event ID 1000의 hgl_loader.exe crash를 캡처한다.
  5. ShimCache/AppCompatCache 또는 collector JSONL에서 C:\Windows\Temp\hgl_loader.exe context를 캡처한다.
  6. Sysmon Event ID 10은 아직 미확인으로 두고, 재현 후 별도 캡처한다.

Final Judgment

결론 SB-07 Page 3는 loader 기반 실행과 dump artifact 흐름을 ELK에서 다시 찾을 수 있음을 보여준다. 다만 hgl_loader.exe -> lsass.exe 직접 ProcessAccess는 아직 미확인이므로, manual mapping은 확인됨이 아니라 manual_mapping_inferred로 분리한다.
